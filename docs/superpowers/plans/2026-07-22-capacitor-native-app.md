# Capacitor Native App Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Wrap the existing PMP Group dashboard (Next.js, server-rendered) as an installable Android app via Capacitor, pointed at the live production URL, with native GPS permission handling for the location-capture feature.

**Architecture:** Capacitor's native shell loads `https://dash.pabrikespmp.com` directly in a WebView (`server.url` config) — no static export, no offline bundle, no changes to how the Next.js app is deployed. The only web-code changes are two small, guarded integration points (GPS, status bar) that detect whether they're running inside the native shell and no-op/fall back to standard browser behavior otherwise, so the same deployed site keeps working identically in desktop browsers, mobile browsers, and the native app.

**Tech Stack:** Capacitor 7.x (`@capacitor/core`, `@capacitor/cli`, `@capacitor/android`, `@capacitor/ios`, `@capacitor/geolocation`, `@capacitor/status-bar`, `@capacitor/assets`), existing Next.js 16 / React 19 / next-themes stack (unchanged).

## Global Constraints

- **No test framework exists in this repo** (no jest/vitest, no `test` script) — every task's verification uses `npx tsc --noEmit`, `npm run lint`, `npm run build`, and Capacitor-CLI-level checks (`npx cap doctor`, `npx cap sync android`) only. Do not add a test framework.
- **This development environment (Windows, no Android SDK/Android Studio, no macOS/Xcode/CocoaPods) cannot compile or run the native apps.** `ANDROID_HOME` is unset and no SDK/Studio install exists on this machine; there is no Mac for iOS tooling either. Every task's verification is scoped to what Node/npm/Capacitor-CLI can check without those toolchains: file existence and content correctness, JSON/TS config validity, `npx cap doctor`, `npx cap sync android` (does not require the SDK, only reads/writes project files), and the existing web-side `tsc`/`lint`/`build`. Actually compiling and installing the APK is a **manual step for the user**, documented in Task 5's handoff doc — do not attempt to invoke Gradle/`assembleDebug` in any task, and do not report a task as failed because an APK wasn't produced.
- **iOS is best-effort scaffolding only, not a required deliverable.** `npx cap add ios` invokes CocoaPods (`pod install`) as part of adding the platform, which is unavailable on Windows and will likely fail. Attempt it in Task 1; if it fails with a CocoaPods/Ruby/Xcode-related error, that is an **expected environment limitation, not a task failure** — skip it, note it in the task report, and continue. Android is the only platform required to succeed in this plan.
- **Production URL:** `https://dash.pabrikespmp.com` — use verbatim in `capacitor.config.ts`.
- **App identity:** `appId: "com.pabrikespmp.dashboard"`, `appName: "PMP Group"` — use verbatim (reverse-domain of `pabrikespmp.com`; approved app-name wording from the design spec).
- **Brand color:** `#0f766e` is used as a concrete hex approximation of this app's CSS `--primary: oklch(0.55 0.14 175)` (teal) for icon/splash generation, since oklch has no single canonical hex equivalent without a color-management library. This is a visual approximation, not a pixel-exact conversion — acceptable per the design spec (no exact hex was mandated there).
- **Package versions:** install with `@latest` at implementation time rather than pinned version numbers (this plan was written without ability to verify current published versions) — record the actually-resolved versions in each task's commit message or report for future reference. Do not hand-edit version numbers into `package.json`; let `npm install` write them.
- Native platform folders (`android/`, and `ios/` if it succeeds) and the `capacitor-www/`/`resources/` folders are **committed to git**, not gitignored — this is standard Capacitor convention, since these folders hold project-specific native config (signing config, manifest entries, generated icons) that isn't safely regenerable from scratch without losing manual tweaks made directly in Android Studio later.
- Design spec: `docs/superpowers/specs/2026-07-22-capacitor-native-app-design.md` — read it for full context; this plan implements it in full except the two refinements noted in Task 1 and Task 4 (dropping `@capacitor/app` as unnecessary, and identifying that status-bar theme sync requires one small web-code touchpoint, same pattern as the already-agreed GPS touchpoint).

- [ ] **Step 1: Write the icon/splash generation script**

This is a one-off build-time utility (not part of the Next.js app or its runtime) that rasterizes the same hexagonal "ice" mark used in `src/components/dashboard/ice-mark.tsx` onto the brand teal background, using `sharp` — already present in `node_modules` as a transitive dependency of this project (verified: `require.resolve("sharp")` resolves cleanly) — so no new runtime dependency is added for this.

Create `scripts/generate-app-icon-assets.mjs`:

```js
// One-off asset-generation script for the Capacitor app icon/splash source
// images. Not part of the Next.js app or its runtime — run manually via
// `node scripts/generate-app-icon-assets.mjs` whenever the brand mark or
// color changes, then re-run `npx capacitor-assets generate --android` to
// regenerate every platform-specific size from its output.
import { mkdir } from "node:fs/promises";
import path from "node:path";
import sharp from "sharp";

const BRAND_TEAL = "#0f766e"; // approximates --primary: oklch(0.55 0.14 175)
const OUT_DIR = path.join(process.cwd(), "resources");

// Same hexagonal "ice" mark as src/components/dashboard/ice-mark.tsx,
// inlined here since this script runs outside the React/Next.js build —
// centered and scaled, white on the brand teal background.
function markSvg({ size, markScale }) {
  const half = size / 2;
  const markSize = size * markScale;
  const offset = half - markSize / 2;
  return `
    <svg width="${size}" height="${size}" viewBox="0 0 ${size} ${size}" xmlns="http://www.w3.org/2000/svg">
      <rect width="${size}" height="${size}" fill="${BRAND_TEAL}" />
      <g transform="translate(${offset}, ${offset}) scale(${markSize / 32})" fill="none" stroke="#ffffff">
        <path d="M16 1 L29 9 V23 L16 31 L3 23 V9 Z" stroke-width="1.6" stroke-linejoin="round" />
        <path d="M16 1 V31 M3 9 L29 23 M29 9 L3 23" stroke-width="1.1" stroke-opacity="0.55" />
        <path d="M16 1 L29 9 V23 L16 31 L3 23 V9 Z" fill="#ffffff" fill-opacity="0.16" />
      </g>
    </svg>
  `;
}

async function main() {
  await mkdir(OUT_DIR, { recursive: true });

  await sharp(Buffer.from(markSvg({ size: 1024, markScale: 0.62 })))
    .png()
    .toFile(path.join(OUT_DIR, "icon.png"));

  await sharp(Buffer.from(markSvg({ size: 2732, markScale: 0.32 })))
    .png()
    .toFile(path.join(OUT_DIR, "splash.png"));

  console.log("Wrote resources/icon.png and resources/splash.png");
}

main();
```

Replace the existing `handleUseMyLocation` function (currently lines 88-111):

```tsx
  function handleUseMyLocation() {
    if (!navigator.geolocation) {
      setGeoError("Perangkat/browser ini tidak mendukung GPS.");
      return;
    }
    setGeoError(null);
    setLocating(true);
    navigator.geolocation.getCurrentPosition(
      (pos) => {
        const { latitude, longitude } = pos.coords;
        onChange({ latitude, longitude, alamat: current.alamat });
        reverseGeocode(latitude, longitude);
        setRecenterKey((k) => k + 1);
        setLocating(false);
      },
      (err) => {
        setGeoError(
          err.code === err.PERMISSION_DENIED ? "Izin akses lokasi ditolak." : "Gagal mengambil lokasi GPS."
        );
        setLocating(false);
      },
      { enableHighAccuracy: true, timeout: 10000 }
    );
  }
```

with:

```tsx
  async function handleUseMyLocation() {
    setGeoError(null);
    setLocating(true);
    try {
      const pos = await Geolocation.getCurrentPosition({ enableHighAccuracy: true, timeout: 10000 });
      const { latitude, longitude } = pos.coords;
      onChange({ latitude, longitude, alamat: current.alamat });
      reverseGeocode(latitude, longitude);
      setRecenterKey((k) => k + 1);
    } catch (err) {
      const message = err instanceof Error ? err.message.toLowerCase() : "";
      setGeoError(message.includes("denied") ? "Izin akses lokasi ditolak." : "Gagal mengambil lokasi GPS.");
    } finally {
      setLocating(false);
    }
  }
```

Why this is safe for the existing web app (not just the native app): `@capacitor/geolocation`'s web implementation transparently falls back to the browser's own `navigator.geolocation` when not running inside a native Capacitor shell — so this same code path continues to work identically in desktop browsers and mobile browsers exactly as it did before, while gaining correct native permission handling when running inside the Android app.

### Task 4: Native status bar theme sync

**Files:**
- Create: `src/components/native-status-bar-sync.tsx`
- Modify: `src/components/providers.tsx`
- Modify: `package.json`, `package-lock.json` (via `npm install`)

**Interfaces:**
- Consumes: `useTheme()` from `next-themes` (already a dependency, already used identically in `src/components/dashboard/appearance-menu.tsx:4,40` — same import, same hook).
- Produces: `NativeStatusBarSync` — a component with no props, rendered once near the app root; nothing else consumes it.

**Design note (refinement from the approved spec):** the spec listed `@capacitor/status-bar` as a native feature without specifying that it requires a small web-code touchpoint. Because this app is loaded via `server.url` (the real, remote Next.js site — not bundled local files), any Capacitor plugin call must happen from *that site's own JS*, the same way the GPS integration in Task 3 does. This task is that touchpoint for the status bar, following the identical pattern: a guarded call that only executes inside the native shell and is a no-op everywhere else.

Separately, the spec also mentioned `@capacitor/app` for Android back-button handling — that is **not needed**: Capacitor's native shell already handles the hardware back button by default (goes back in WebView history if possible, otherwise backgrounds/exits the app), with no plugin or JS code required. This task does not install `@capacitor/app`.

- [ ] **Step 2: Create the sync component**

Create `src/components/native-status-bar-sync.tsx`:

```tsx
"use client";

import { useEffect } from "react";
import { useTheme } from "next-themes";
import { Capacitor } from "@capacitor/core";
import { StatusBar, Style } from "@capacitor/status-bar";

// Keeps the native status bar icon color in sync with the app's light/dark
// theme when running inside the Capacitor native shell. Because this app is
// loaded via server.url (the real deployed site, not bundled local files),
// this is the one place status-bar styling can be driven from — it no-ops
// entirely in a regular browser, where StatusBar has no meaning.
export function NativeStatusBarSync() {
  const { resolvedTheme } = useTheme();

  useEffect(() => {
    if (!Capacitor.isNativePlatform()) return;
    StatusBar.setStyle({ style: resolvedTheme === "dark" ? Style.Dark : Style.Light }).catch(() => {});
  }, [resolvedTheme]);

  return null;
}
```

```markdown
# Build & Install the Android App

The Capacitor project in this repo (`android/`) cannot be compiled in the
development environment these changes were written in (no Android SDK
installed). This is the manual process to actually produce and install the
APK, done once you have Android Studio available.

## Prerequisites

- [Android Studio](https://developer.android.com/studio) installed (it
  bundles the Android SDK).
- This repo cloned, with `npm install` already run.

## One-time: create a release signing key

Only needed the first time, or if the key is ever lost (losing it means you
can never publish an update under the same app identity again — back it up
somewhere safe, outside this git repo):

```bash
keytool -genkey -v -keystore pmp-release.keystore -alias pmp-release -keyalg RSA -keysize 2048 -validity 10000
```

Answer its prompts (name, org, etc. — cosmetic, not user-facing) and choose
a strong password. Then create `android/key.properties` (already
`.gitignore`d — never commit this file):

```properties
storeFile=../../pmp-release.keystore
storePassword=<the password you set above>
keyAlias=pmp-release
keyPassword=<the password you set above>
```

## Build the APK

1. Open the `android/` folder in Android Studio (File > Open).
2. Let it finish Gradle sync (first time may take several minutes while it
   downloads the Gradle distribution and dependencies — needs internet).
3. Build > Generate Signed Bundle / APK... > APK > select the keystore from
   the previous step > Release > Finish.
4. The signed APK appears under `android/app/release/app-release.apk`.

## Install & test on a device

1. Enable "Install unknown apps" for whichever app you'll transfer the APK
   through (e.g. Files, a messaging app) on the target Android phone —
   Settings > Apps > Special access > Install unknown apps.
2. Transfer `app-release.apk` to the phone and open it to install.
3. Open the app and verify:
   - It loads the dashboard and the login screen appears.
   - Logging in works and the session persists after fully closing and
     reopening the app.
   - Navigate to Pemasaran > "+ Pengajuan Baru" and tap "Pakai Lokasi
     Saya" in the Lokasi GPS section — Android should show its standard
     location-permission dialog the first time; after granting it, the map
     pin should move to the device's actual GPS position.
   - The hardware Back button navigates back through the app's pages, and
     exits/backgrounds the app from the home/root page.
   - Toggling light/dark mode (Mode Tampilan menu) changes the status bar
     icon color to match.

## Updating the app later

Whenever the web app at `https://dash.pabrikespmp.com` changes, **no
rebuild is needed** — the app loads that URL live, so changes appear the
next time the app is opened or the page reloads. A new APK build is only
needed when this repo's `android/`, `capacitor.config.ts`, or native plugin
set changes (e.g. adding a new Capacitor plugin).
```
